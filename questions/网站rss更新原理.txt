对 MkDocs 这类**静态站点生成器**来说，所谓“更新 RSS”其实只有两步，**全程发生在本地构建的一两秒内**，之后只是把生成的 XML 文件随 HTML 一起上传。具体原理如下：

1. 读取全部文章元数据  
   插件在 `on_post_build` 阶段扫描整个 `nav`，把每个页面的 Front-matter（`title`、`date`、`description`…）读进内存，并按 `date` 做倒序排序，挑出最新 N 条（默认 20，可在 `mkdocs.yml` 里用 `length` 调）。

2. 拼 XML 字符串  
   拿现成的 Jinja2 模板（插件内置 `rss.xml.j2`）做渲染，把站点标题、描述、首页 URL、构建时间(`lastBuildDate`) 以及上一步得到的条目列表一次性拼成符合 RSS 2.0 规范的 XML 文本。

3. 写入输出目录  
   渲染完的字符串直接写成 `site/feed_rss_created.xml`（文件名可配置）。  
   整个过程**不依赖 Git 时间戳，也不轮询远程接口**，就是本地一次性把“当前最新”状态写死到文件里；你下次再跑 `mkdocs build`，它会重新扫描、重新排序、重新生成，旧的 XML 直接被覆盖。

4. 上传即“更新”  
   你把 `site/` 里所有文件（HTML + XML + 资源）推到 Web 服务器后，阅读器再来抓 `feed_rss_created.xml`，发现里面有新条目，于是认为“网站更新了”。实际上服务端只是换了个静态文件，没有任何动态脚本或数据库参与。

总结：  
MkDocs 的 RSS “更新”就是**每次构建时重新生成一份静态 XML**，把最新文章信息写进去；阅读器靠定期拉取这份文件、比对 `<guid>`/`<pubDate>` 来判断有没有新内容，整个机制纯靠**静态文件 + 阅读器轮询**，没有主动推送，也没有后台守护进程。
